# Localization (i18n)

OpenHosting's interface can be translated. It ships with **English** and
**Dutch**, and a language picker appears in the storefront header.

## For operators

When more than one language is available, customers pick their language from the
storefront header; the choice is remembered in a cookie and applied across the
storefront, authentication pages, client area and admin navigation.

## How it works

Translations are plain dictionaries — no build step, no external service. Each
UI string has a dot-path key (e.g. `nav.cart`, `dash.welcome`) mapped to its
translation. The active locale is chosen by cookie, with English as the
fallback for any missing key.

The dictionaries and the translate helper live in
[`src/lib/i18n.ts`](../../src/lib/i18n.ts). Server components call `getT()` to
get a translate function bound to the request's locale.

## Adding a language

1. Open `src/lib/i18n.ts`.
2. Copy the `nl` (Dutch) dictionary object and translate each value to your
   language.
3. Register it in `DICTIONARIES` and add it to `LOCALES` with a display label.

The language picker picks it up automatically. Because every key falls back to
English, a partial translation is fine — untranslated strings simply show in
English until you fill them in.

## Translating a new page

As pages are migrated to the translation system, they call `getT()` and
reference message keys instead of hardcoded strings. If you add UI, follow the
same pattern: add keys to the dictionaries and use `t("your.key")`. Contributions
that extend coverage to more pages are welcome — see
[Contributing](../contributing.md).
